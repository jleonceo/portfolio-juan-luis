Objetivo del proyecto

Construir modelos de ML (scikit-learn) sobre 3 años de datos históricos SQL para predecir ventas a 8 días vista, a nivel tienda-producto. El sistema final corre en batch periódico. El impacto: reducir costes de almacén y capital, y aumentar ingresos reduciendo roturas de stock.

Arquitectura del pipeline

1

Extracción de datos (SQL)

Base de datos con 3 años de histórico de ventas por tienda y producto. Categoría: alimentación.

2

Preprocesamiento y feature engineering

Lags, variables temporales, variables sintéticas para demanda intermitente, tratamiento de ceros.

3

Modelado ML (scikit-learn / LightGBM)

Un modelo por SKU o a nivel jerárquico superior. Estrategia directa o recursiva para horizonte multi-día.

4

Reconciliación jerárquica

Asegurar coherencia entre predicciones a nivel categoría, subcategoría y producto.

5

Sistema batch de predicción

Despliegue periódico que genera el forecast de los próximos 8 días.

Problemas avanzados y cómo resolverlos

Forecasting jerárquico Problema

Diferentes niveles en el catálogo (categoría → subcategoría → producto). Las predicciones probabilísticas no coinciden entre niveles.

Bottom-up Top-down Híbrida Reconciliación

Demanda intermitente Problema

Días con ventas = 0 sin causa clara (rotura de stock o demanda real cero). Genera ruido en el modelo.

Marcas de rotura de stock Variables sintéticas Método Croston Modelo prob. de cero

Cientos / miles de SKUs Problema

Escalar a miles de productos es computacionalmente intensivo si se entrena un modelo por SKU de forma clásica.

LightGBM Top-down sin modelos inferiores Cloud / clusters

Predicción multipaso (8 días) Problema

Los modelos ML predicen registro a registro; para horizontes multi-día necesitas info de futuro no disponible.

Estrategia directa (N modelos) Estrategia recursiva (1 modelo) Recursiva = más fácil de mantener
